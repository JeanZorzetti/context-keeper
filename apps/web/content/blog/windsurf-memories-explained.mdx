## What Are Windsurf Memories?

**Windsurf Memories is a built-in feature of Windsurf's Cascade AI that allows the agent to remember information across sessions — project preferences, architectural decisions, naming conventions, and working patterns.** Unlike a static rules file you write and maintain, Windsurf Memories is semi-dynamic: Windsurf can suggest memories to save based on your conversations, and you can manually add or edit them.

This addresses the core problem of [context rot](/blog/what-is-context-rot-in-ai-coding-agents) — the loss of project knowledge between sessions. Every time a new Cascade session starts, it has access to saved memories rather than starting completely cold.

## How Windsurf Memories Work

Windsurf Memories operates through three mechanisms:

**1. Automatic suggestions.** During a session, when Cascade detects that you've established a significant pattern or preference, it may suggest saving it to Memory: "Should I remember that we use Prisma with PostgreSQL for this project?"

**2. Manual saves.** You can explicitly ask Cascade to remember something:
```
Remember that we're using Tailwind CSS only — no CSS Modules or styled-components.
```

**3. Direct editing.** You can view, edit, add, and delete memories through Windsurf's Memory panel (Settings → AI → Memories).

When Cascade starts a new session, it has access to all saved memories and uses them to inform its responses.

## Types of Information to Store in Memories

The most valuable things to save in Windsurf Memories:

**Tech stack specifics:**
- "This project uses Next.js 15 App Router, not Pages Router"
- "TypeScript strict mode is enabled — no any types"
- "Testing: Vitest, not Jest"

**Architecture decisions:**
- "We use Prisma for database access — no raw SQL"
- "Auth is handled by Auth0, not NextAuth"
- "Error handling: return objects with data/error fields, don't throw"

**Naming and file conventions:**
- "API routes go in app/api/ — route handlers, not page-based API routes"
- "Components use PascalCase filenames in components/[domain]/"
- "Tests co-located with source files as [filename].test.ts"

**Explicit rejections:**
- "We rejected tRPC — use plain Next.js route handlers"
- "Don't use Redux — Zustand for state management"

**Personal preferences:**
- "Always run tests after making changes to the auth module"
- "When suggesting code, prefer explicit types over inference for function parameters"

## Setting Up Windsurf Memories for a New Project

When starting a new project in Windsurf, front-load memory creation to prevent the cold-start problem:

**Step 1: Open a new Windsurf session in your project directory.**

**Step 2: Tell Cascade to build initial memories:**

```
I'm starting a new project. Here's the context you should remember:

Tech stack: Next.js 15.3 App Router, TypeScript strict mode, PostgreSQL via Prisma 7, 
Auth0 v3, Tailwind CSS 3.4, Vitest for testing.

Architecture decisions:
- Error handling: return { data } or { error: string } — never throw from API handlers
- Auth: Auth0 (not NextAuth, not Passport)
- Database: Prisma only (no raw SQL, no other ORM)
- API: Next.js route handlers in app/api/ (no tRPC)

Do not use:
- localStorage for auth state
- Class components (hooks only)
- The Pages Router

Save these to Memory so you have them in future sessions.
```

Cascade will create a set of memories capturing these decisions. Each subsequent session starts with this context already loaded.

**Step 3: Verify by checking the Memory panel** (Settings → AI → Memories). You should see the entries you just created.

## Managing and Maintaining Memories

### Regular Memory Audits

Memories can become stale. If you switch libraries, change patterns, or make new architectural decisions, old memories may conflict with new ones. Schedule a monthly memory review:

1. Open the Memories panel
2. Check for outdated entries (old library versions, rejected patterns)
3. Update or delete stale memories
4. Add new decisions made in recent sessions

A memory that says "use Redux for state" when you've since migrated to Zustand will actively mislead Cascade.

### Keeping Memories Concise

Each memory should be one specific fact or decision. Avoid memory entries that bundle multiple ideas:

**Bad:** "The project uses Next.js App Router with TypeScript and we prefer Tailwind over CSS Modules and we're using Prisma for the database."

**Better — three separate memories:**
1. "Framework: Next.js 15 App Router (not Pages Router)"
2. "Styling: Tailwind CSS — no CSS Modules or styled-components"
3. "Database: Prisma 7 with PostgreSQL — no raw SQL"

Concise, single-topic memories are easier to maintain and less likely to have conflicts.

## Windsurf Memories vs CLAUDE.md / AGENTS.md

Both solve context persistence, but through very different mechanisms:

| | Windsurf Memories | CLAUDE.md |
|---|---|---|
| **Format** | AI-managed entries | Markdown file |
| **Version control** | No | Yes (committed to repo) |
| **Team sharing** | No (personal per-user) | Yes (everyone gets it) |
| **Portability** | Windsurf only | Works in Claude Code, Cursor, etc. |
| **Maintenance** | Semi-automatic | Manual (or auto-captured) |
| **Visibility** | Windsurf Memory panel | Any text editor / git log |

**For Windsurf-only teams:** Memories work well — automatic suggestions reduce maintenance burden and the context persists reliably.

**For multi-tool teams** (Windsurf + Claude Code + Cline): You need a tool-agnostic solution. `CLAUDE.md` or `AGENTS.md` is version-controlled, portable, and works with all AI coding tools. Keep this as the canonical source and let Memories supplement it.

A good hybrid approach:
1. Maintain `AGENTS.md` in your project for the canonical architectural decisions
2. Use Windsurf Memories for Windsurf-specific preferences and session-specific learnings
3. Keep the two roughly in sync on major decisions

## Windsurf Memories Limitations

**They're personal, not team-shared.** Each developer's Windsurf Memories are their own. A new team member's Windsurf starts with no project memories. For team context, you need a version-controlled file.

**They're Windsurf-only.** If you switch to Cursor or Claude Code for a session, the Windsurf Memories aren't available. [Context rot](/blog/what-is-context-rot-in-ai-coding-agents) returns unless you have a parallel `CLAUDE.md` or `.cursorrules`.

**Semi-automatic isn't fully automatic.** Windsurf may suggest memories but doesn't guarantee it captures every significant decision. Important decisions made mid-session may not get suggested for memory. Explicit manual saves are often necessary.

## Key Takeaways

- Windsurf Memories = Cascade's persistent context store; semi-automatic (suggests + manual saves)
- Most valuable content: tech stack decisions, architecture decisions, explicit rejections, naming conventions
- Front-load memory creation on new projects to prevent the cold-start problem
- Audit memories monthly — stale entries actively mislead the AI
- Keep memories concise and single-topic for easier maintenance
- Memories are personal (per-developer) and Windsurf-only — not a substitute for `AGENTS.md` in multi-tool teams
- Pair Memories with `AGENTS.md` for a robust multi-tool context strategy
